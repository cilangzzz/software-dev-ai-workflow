# 架构设计用例

## 用例1：电商系统架构设计

### 输入
```
需求：开发一个B2C电商平台，支持用户注册登录、商品浏览、购物车、订单管理、
支付功能。预计日均PV 100万，高峰期QPS 5000。
约束：团队熟悉Java和MySQL，预算有限，希望在3个月内上线MVP。
```

### 输出
```markdown
# 电商系统架构设计

## 1. 架构概述
采用微服务架构，前后端分离，支持水平扩展。初期采用单体优先策略，
预留微服务拆分接口，随业务发展逐步演进。

## 2. 技术选型

| 层级 | 技术选择 | 选择理由 |
|------|---------|---------|
| 前端 | Vue 3 + Vite | 团队熟悉，生态完善，性能优秀 |
| 后端 | Spring Boot 3 | Java生态，团队熟悉，成熟稳定 |
| 数据库 | MySQL 8.0 | 关系型数据库，团队熟悉，成本低 |
| 缓存 | Redis | 高性能缓存，支持分布式Session |
| 消息队列 | RabbitMQ | 轻量级，成本低，满足当前需求 |
| 网关 | Spring Cloud Gateway | 统一入口，限流熔断 |
| 容器 | Docker + Docker Compose | 简化部署，初期无需K8s |

## 4. 模块划分

### 4.1 核心服务
- **用户服务(user-service)**
  - 用户注册、登录、权限管理
  - 用户信息管理
  - 收货地址管理

- **商品服务(goods-service)**
  - 商品管理、分类、搜索
  - 库存管理
  - 商品评价

- **订单服务(order-service)**
  - 订单创建、状态管理
  - 购物车
  - 订单查询

- **支付服务(payment-service)**
  - 支付对接（微信、支付宝）
  - 支付状态回调
  - 退款处理

### 4.2 公共模块
- **common-core**: 公共工具类、异常处理、统一响应
- **common-redis**: Redis配置和工具类
- **common-security**: 安全认证组件（JWT）
- **common-log**: 日志组件

## 5. 数据库设计

### 核心表结构
```sql
-- 用户表
CREATE TABLE users (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    username VARCHAR(50) UNIQUE NOT NULL,
    password VARCHAR(255) NOT NULL,
    email VARCHAR(100),
    phone VARCHAR(20),
    status TINYINT DEFAULT 1,
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    updated_at DATETIME ON UPDATE CURRENT_TIMESTAMP
);

-- 商品表
CREATE TABLE goods (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    name VARCHAR(200) NOT NULL,
    category_id BIGINT,
    price DECIMAL(10,2) NOT NULL,
    stock INT DEFAULT 0,
    status TINYINT DEFAULT 1,
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP
);

-- 订单表
CREATE TABLE orders (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    order_no VARCHAR(50) UNIQUE NOT NULL,
    user_id BIGINT NOT NULL,
    total_amount DECIMAL(10,2),
    status TINYINT DEFAULT 0,
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP
);
```

## 6. 风险评估

| 风险 | 影响 | 概率 | 缓解措施 |

| 服务雪崩 | 高 | 中 | 引入熔断降级、限流 |
| 数据一致性 | 高 | 中 | 最终一致性、消息补偿 |
| 团队经验不足 | 中 | 高 | 技术培训、代码审查 |

## 用例3：ADR示例 - 数据库选型

### 输入
```
创建ADR：项目需要选择主数据库，团队在MySQL和PostgreSQL之间犹豫。
我们需要JSON支持、全文搜索、高并发场景下表现好。
```

### 输出
```markdown
# ADR-001: 选择PostgreSQL作为主数据库

## 状态
Accepted

## 背景
新开发的电商平台需要一个关系型数据库，主要需求包括：
1. 支持复杂查询和事务
2. 原生JSON支持（商品属性灵活存储）
3. 全文搜索能力（商品搜索）
4. 良好的性能和可扩展性
5. 团队有运维经验

## 决策
选择PostgreSQL作为主数据库。

## 理由
1. **JSONB性能优秀**: 比MySQL的JSON类型性能更好，适合商品灵活属性
2. **内置全文搜索**: 可暂时不引入Elasticsearch，降低系统复杂度
3. **MVCC实现优秀**: 并发控制更好，适合高并发场景
4. **查询优化器**: 复杂查询性能更优
5. **开源免费**: 无License成本

## 备选方案

### 方案1: MySQL 8.0
- 优点: 团队更熟悉，生态更成熟
- 缺点: JSON性能较弱，全文搜索有限
- 结论: 不满足JSON和搜索需求

### 方案2: MongoDB
- 优点: 天然JSON支持，灵活Schema
- 缺点: 事务支持有限，报表查询复杂
- 结论: 不适合电商事务场景

## 影响
### 正面影响
- 商品属性存储更灵活
- 可暂时不引入ES
- 复杂查询性能更好

### 负面影响
- 团队需要学习PostgreSQL特性
- 云数据库选型较少
- 部分ORM工具支持不如MySQL

## 风险与缓解
| 风险 | 缓解措施 |
|------|---------|
| 团队经验不足 | 安排培训，编写最佳实践文档 |
| 运维复杂度 | 使用云数据库服务 |

## 参考
- PostgreSQL官方文档
- MySQL vs PostgreSQL性能对比报告
```
