# 架构设计用例

## 用例1：电商系统架构设计

### 场景描述
架构师需要为一个新的B2C电商平台设计系统架构，需要考虑性能、成本和团队技术栈。

### 使用Skill
- **Skill**: architect
- **触发方式**: `/architect` 或自然语言 "设计系统架构"

### 完整提示词
```
/architect

需求：开发一个B2C电商平台，支持用户注册登录、商品浏览、购物车、订单管理、
支付功能。预计日均PV 100万，高峰期QPS 5000。
约束：团队熟悉Java和MySQL，预算有限，希望在3个月内上线MVP。
```

### 输入参数
| 参数名 | 类型 | 必填 | 值 |
|--------|------|------|-----|
| requirements | string | 是 | "开发一个B2C电商平台，支持用户注册登录、商品浏览、购物车、订单管理、支付功能" |
| constraints | string | 否 | "团队熟悉Java和MySQL，预算有限，3个月内上线MVP" |
| preferences | string | 否 | "Java, MySQL" |
| scale | string | 否 | "medium" |

### 执行过程

#### 步骤1：需求分析
解析需求中的功能和非功能需求：

**功能需求**：
- 用户注册登录
- 商品浏览
- 购物车
- 订单管理
- 支付功能

**非功能需求**：
- 日均PV: 100万
- 高峰期QPS: 5000
- 上线时间: 3个月

#### 步骤2：技术选型
推荐技术栈方案：

| 层级 | 选择 | 理由 |
|------|------|------|
| 前端 | Vue 3 + Vite | 团队熟悉，生态完善，性能优秀 |
| 后端 | Spring Boot 3 | Java生态，团队熟悉，成熟稳定 |
| 数据库 | MySQL 8.0 | 关系型数据库，团队熟悉，成本低 |
| 缓存 | Redis | 高性能缓存，支持分布式Session |
| 消息队列 | RabbitMQ | 轻量级，成本低，满足当前需求 |
| 容器 | Docker + Docker Compose | 简化部署，初期无需K8s |

#### 步骤3：架构设计
生成系统架构图和模块划分，采用微服务架构，前后端分离，支持水平扩展。

#### 步骤4：决策记录
自动生成ADR文档：
- ADR-001: 选择Spring Boot作为后端框架
- ADR-002: 使用Redis作为分布式缓存
- ADR-003: 采用单体优先策略，预留微服务拆分接口

#### 步骤5：风险评估
识别架构风险点：

| 风险 | 影响 | 概率 | 缓解措施 |
|------|------|------|---------|
| 单体数据库瓶颈 | 高 | 中 | 读写分离、预留分库分表接口 |
| 服务雪崩 | 高 | 中 | 引入熔断降级、限流 |
| 数据一致性 | 高 | 中 | 最终一致性、消息补偿 |
| 团队经验不足 | 中 | 高 | 技术培训、代码审查 |

#### 步骤6：方案对比
提供备选方案及权衡分析：

**微服务 vs 单体**
| 维度 | 微服务 | 单体 |
|------|-------|-------|
| 开发效率 | 低（复杂度高） | 高（简单直接） |
| 性能 | 高（可独立扩展） | 中 |
| 可维护性 | 高（解耦） | 中 |
| 学习成本 | 高 | 低 |

**结论**: 采用单体优先策略，预留微服务拆分接口。

# ADR-001: 选择PostgreSQL作为主数据库

## 状态
Accepted

## 背景
新开发的电商平台需要一个关系型数据库，主要需求包括：
1. 支持复杂查询和事务
2. 原生JSON支持（商品属性灵活存储）
3. 全文搜索能力（商品搜索）
4. 良好的性能和可扩展性
5. 团队有运维经验

## 决策
选择PostgreSQL作为主数据库。

## 理由
1. **JSONB性能优秀**: 比MySQL的JSON类型性能更好，适合商品灵活属性
2. **内置全文搜索**: 可暂时不引入Elasticsearch，降低系统复杂度
3. **MVCC实现优秀**: 并发控制更好，适合高并发场景
4. **查询优化器**: 复杂查询性能更优
5. **开源免费**: 无License成本

## 备选方案

### 方案1: MySQL 8.0
- 优点: 团队更熟悉，生态更成熟
- 缺点: JSON性能较弱，全文搜索有限
- 结论: 不满足JSON和搜索需求

### 方案2: MongoDB
- 优点: 天然JSON支持，灵活Schema
- 缺点: 事务支持有限，报表查询复杂
- 结论: 不适合电商事务场景

## 影响
### 正面影响
- 商品属性存储更灵活
- 可暂时不引入ES
- 复杂查询性能更好

### 负面影响
- 团队需要学习PostgreSQL特性
- 云数据库选型较少
- 部分ORM工具支持不如MySQL

## 风险与缓解
| 风险 | 缓解措施 |
|------|---------|
| 团队经验不足 | 安排培训，编写最佳实践文档 |
| 运维复杂度 | 使用云数据库服务 |

## 参考
- PostgreSQL官方文档
- MySQL vs PostgreSQL性能对比报告
