# 需求评审用例

## 用例1：订单退款功能评审

### 场景描述
现有电商平台需要新增订单退款功能，研发团队需要对需求进行评审，分析对现有设计的影响。

### 使用Skill
- **Skill**: requirement-review
- **触发方式**: `/requirement-review` 或自然语言 "评审需求"

### 完整提示词
```
/requirement-review

需求文档：订单退款功能
1. 用户可在订单完成后30天内申请退款
2. 支持全额退款和部分退款
3. 退款申请提交后进入商家审核
4. 审核通过后自动原路退款
5. 用户可查看退款进度

业务规则：
- 订单状态为"已完成"才可申请退款
- 部分退款时，按商品实付金额计算
- 退款金额不超过订单实付金额

评审深度：standard
```

### 输入参数
| 参数名 | 类型 | 必填 | 值 |
|--------|------|------|-----|
| requirement_doc | string | 是 | "订单退款功能需求文档" |
| data_model_doc | string | 否 | "docs/design/data-model.md" |
| domain_design_doc | string | 否 | "docs/design/domain-design.md" |
| database_design_doc | string | 否 | "docs/design/database-design.md" |
| review_depth | string | 否 | "standard" |

### 执行过程

#### 步骤1：需求解析
解析需求文档，提取功能点、实体、业务规则：

**功能点**：
- 用户申请退款（全额/部分退款）
- 商家审核退款申请
- 退款进度查询
- 退款到账通知

**实体识别**：
- RefundOrder（退款订单）
- RefundItem（退款商品明细）

**业务规则**：
- 退款时间限制：订单完成后30天内
- 退款条件：订单状态为"已完成"
- 退款金额限制：不超过订单实付金额

#### 步骤2：文档加载
加载现有设计文档：
- 数据模型：Order、OrderItem、User、Product、Payment
- 领域设计：订单域、支付域
- 数据库设计：orders、order_items表

#### 步骤3：实体映射
将需求中的实体与现有数据模型进行匹配：

| 需求实体 | 现有实体 | 匹配结果 |
|----------|----------|----------|
| RefundOrder | 无 | 需新增 |
| Order.refundStatus | Order | 需扩展属性 |
| User | User | 完全复用 |
| Payment | Payment | 需扩展退款流水 |

#### 步骤4：影响分析
分析需求对现有设计的影响范围：

**新增实体**：RefundOrder、RefundItem
**修改实体**：Order（新增refundStatus、refundAmount）
**数据库变更**：新增refund_order、refund_item表，修改orders表

#### 步骤5：冲突检测
识别需求与现有设计的潜在冲突：
- ✅ 无明显设计冲突
- ⚠️ 需确认部分退款库存处理规则
- ⚠️ 需确认大额退款风控方案

#### 步骤6：可行性评估
评估技术可行性和实现复杂度：

| 维度 | 评估结果 |
|------|----------|
| 实现复杂度 | ⭐⭐⭐ 中等 |
| 技术成熟度 | ⭐⭐⭐⭐⭐ 成熟 |
| 团队能力 | ⭐⭐⭐⭐ 匹配 |

#### 步骤7：风险识别
识别技术风险和业务风险：
- 🔴 高风险：退款与支付回调并发、大额退款资金安全
- 🟠 中风险：部分退款库存回滚、历史订单表结构变更

#### 步骤8：报告生成
生成需求评审报告。

### 最终输出
```markdown
# 需求评审报告

## 评审概要
- 需求名称: 订单退款功能
- 评审时间: 2024-01-15 10:30:00
- 评审深度: standard
- 评审结论: ⚠️ 有条件通过

## 需求摘要
实现订单退款功能，支持用户在订单完成后申请退款，包括：
1. 用户申请退款（全额/部分退款）
2. 商家审核退款申请
3. 退款进度查询
4. 退款到账通知

## 实体影响分析

### 新增实体
| 实体名称 | 来源字段 | 建议的数据模型位置 |
|----------|----------|-------------------|
| RefundOrder | orderNo, refundAmount, reason, status | 订单域-退款聚合 |

### 修改实体
| 实体名称 | 变更类型 | 新增属性 | 影响范围 |
|----------|----------|----------|----------|
| Order | 属性扩展 | refundStatus, refundAmount | 订单查询、订单列表 |

### 复用实体
| 实体名称 | 复用方式 | 现有属性满足度 |
|----------|----------|---------------|
| User | 关联引用 | ✅ 完全满足 |
| Payment | 关联引用 | ⚠️ 部分满足（需新增退款流水记录） |

## 数据库影响分析

### 表结构变更
| 变更类型 | 表名 | 变更详情 | 影响等级 |
|----------|------|----------|----------|
| 新增 | refund_order | 退款订单主表 | 🟠 中 |
| 新增 | refund_item | 退款商品明细表 | 🟠 中 |
| 修改 | orders | 新增refund_status, refund_amount字段 | 🔴 高 |
| 修改 | payment_record | 新增refund_trace_id字段 | 🟢 低 |

### 数据迁移需求
| 迁移场景 | 涉及表 | 数据量预估 | 复杂度 |
|----------|--------|-----------|--------|
| 历史订单退款状态初始化 | orders | 500万+ | 中 |

## 风险识别

### 🔴 高风险
| 风险描述 | 影响范围 | 缓解措施建议 |
|----------|----------|--------------|
| 退款与支付回调并发 | 可能导致状态不一致 | 使用分布式锁+状态机 |
| 大额退款资金安全 | 财务风险 | 增加风控审核环节 |

### 🟠 中风险
| 风险描述 | 影响范围 | 缓解措施建议 |
|----------|----------|--------------|
| 部分退款库存回滚 | 库存数据准确性 | 明确部分退款库存处理规则 |
| 历史订单表结构变更 | 500万+数据迁移 | 分批迁移，灰度验证 |

## 待澄清问题

| # | 问题 | 优先级 | 提问对象 | 状态 |
|---|------|--------|----------|------|
| 1 | 部分退款时，已使用的优惠券如何处理？ | P0 | 产品 | 待确认 |
| 2 | 退款审核是否需要多级审核？ | P0 | 产品/业务 | 待确认 |
| 3 | 退款超时未审核如何处理？ | P1 | 产品 | 待确认 |
| 4 | 是否支持多次部分退款？上限是多少？ | P1 | 产品 | 待确认 |

## 评审结论

### 总体评价
需求清晰，技术方案可行。主要风险在于退款与支付的并发处理、大额退款的资金安全。

### 前置条件
1. 确认部分退款时优惠券处理规则
2. 确认退款审核流程
3. 制定大额退款风控方案

### 下一步
1. 产品确认待澄清问题后，进入技术方案设计阶段
2. 调用 `/architect` 生成退款模块详细技术方案
3. 调用 `/adr` 记录退款流程关键决策
```

## 用例2：会员积分系统评审

需要评审会员积分系统需求，重点关注数据量增长和性能问题。

- **Skill**: requirement-review
- **触发方式**: `/requirement-review` 或自然语言 "分析这个需求的影响"

### 完整提示词
```
/requirement-review

需求文档：会员积分系统
1. 用户购物获得积分（1元=1积分）
2. 积分可用于抵扣现金（100积分=1元）
3. 积分有效期2年，到期自动清零
4. 积分明细可查询
5. 特殊活动期间积分翻倍

评审深度：deep
关注领域：data, performance
```

### 输入参数
| 参数名 | 类型 | 必填 | 值 |
|--------|------|------|-----|
| requirement_doc | string | 是 | "会员积分系统需求" |
| data_model_doc | string | 否 | "docs/design/data-model.md" |
| database_design_doc | string | 否 | "docs/design/database-design.md" |
| review_depth | string | 否 | "deep" |
| focus_areas | string | 否 | "data, performance" |

### 执行过程

#### 步骤1：需求解析
解析需求文档，提取功能点、实体、业务规则：

**功能点**：
- 积分获取（购物获得）
- 积分抵扣
- 积分过期处理
- 积分明细查询
- 活动积分翻倍

**实体识别**：
- MemberPoints（积分账户）
- PointsRecord（积分明细）
- PointsRule（积分规则）

#### 步骤2：文档加载
加载现有设计文档，分析现有数据模型。

#### 步骤3：实体映射
将需求中的实体与现有数据模型进行匹配：

| 需求实体 | 现有实体 | 匹配结果 |
|----------|----------|----------|
| MemberPoints | 无 | 需新增 |
| PointsRecord | 无 | 需新增 |
| User.points | User | 需扩展属性 |
| Order.points | Order | 需扩展属性 |

#### 步骤4：影响分析
分析需求对现有设计的影响范围：

**重点关注**：
- 积分明细表数据量预估：年增长5000万+，3年预估1.5亿+
- 高频查询：用户积分查询（QPS 500+）

#### 步骤5：冲突检测
识别需求与现有设计的潜在冲突：
- ✅ 无明显设计冲突
- ⚠️ 积分明细表数据量增长需重点关注

#### 步骤6：可行性评估
评估技术可行性和实现复杂度：

| 维度 | 评估结果 |
|------|----------|
| 实现复杂度 | ⭐⭐⭐ 中等 |
| 性能挑战 | ⭐⭐⭐⭐ 较高 |
| 团队能力 | ⭐⭐⭐⭐ 匹配 |

#### 步骤7：风险识别
识别技术风险和业务风险：
- 🔴 高风险：积分明细表数据量爆炸、积分并发扣减、积分过期批量处理
- 🟠 中风险：积分规则动态变更、积分抵扣计算精度

#### 步骤8：报告生成
生成需求评审报告，重点关注数据量和性能分析。

### 最终输出
```markdown
# 需求评审报告

## 评审概要
- 需求名称: 会员积分系统
- 评审时间: 2024-01-15 14:00:00
- 评审深度: deep
- 关注领域: data, performance
- 评审结论: ⚠️ 有条件通过

## 实体影响分析

### 新增实体
| 实体名称 | 来源字段 | 建议的数据模型位置 |
|----------|----------|-------------------|
| MemberPoints | userId, totalPoints, availablePoints | 会员域-积分聚合 |
| PointsRecord | userId, points, type, source, expireAt | 会员域-积分聚合 |
| PointsRule | ruleId, ratio, startTime, endTime | 会员域-规则聚合 |

## 数据库影响分析

### 表结构变更
| 变更类型 | 表名 | 变更详情 | 影响等级 |
|----------|------|----------|----------|
| 新增 | member_points | 用户积分账户表 | 🟠 中 |
| 新增 | points_record | 积分明细表（流水） | 🔴 高 |
| 新增 | points_rule | 积分规则表 | 🟢 低 |

### 性能分析

#### 数据量预估
| 表 | 年增长量 | 3年预估 | 索引策略 |
|----|----------|---------|----------|
| points_record | 5000万+ | 1.5亿+ | 分区表 + 冷热分离 |

#### 性能优化建议
1. **积分明细表分区**: 按月份分区，提升查询效率
2. **积分账户缓存**: Redis缓存用户积分余额
3. **过期检查异步化**: 使用消息队列异步处理积分过期
4. **读写分离**: 积分明细查询走从库

## 风险识别

### 🔴 高风险
| 风险描述 | 影响范围 | 缓解措施建议 |
|----------|----------|--------------|
| 积分明细表数据量爆炸 | 查询性能、存储成本 | 分区表+冷热分离+归档策略 |
| 积分并发扣减 | 积分超扣 | Redis分布式锁+乐观锁 |
| 积分过期批量处理 | 数据库压力 | 分批处理+异步化 |

## 待澄清问题

| # | 问题 | 优先级 | 提问对象 | 状态 |
|---|------|--------|----------|------|
| 1 | 积分过期是按获得时间还是统一年末？ | P0 | 产品/业务 | 待确认 |
| 2 | 积分抵扣是否支持部分使用？ | P0 | 产品 | 待确认 |
| 3 | 退款时积分如何处理？ | P1 | 产品 | 待确认 |

## 评审结论

### 前置条件
1. 确认积分过期规则
2. 确认积分抵扣使用规则
3. 制定积分明细数据归档策略

### 下一步
1. 确认待澄清问题后，调用 `/architect` 生成详细技术方案
2. 调用 `/adr` 记录积分系统关键决策
```

## 用例4：与现有设计冲突检测

### 场景描述
需求设计与现有设计存在冲突，需要检测并提出解决方案。

### 使用Skill
- **Skill**: requirement-review
- **触发方式**: `/requirement-review` 或自然语言 "检查需求与现有设计的兼容性"

### 完整提示词
```
/requirement-review

需求：需要新增一个商品库存表，包含商品ID和库存数量

评审深度：standard
```

### 输入参数
| 参数名 | 类型 | 必填 | 值 |
|--------|------|------|-----|
| requirement_doc | string | 是 | "需要新增一个商品库存表，包含商品ID和库存数量" |
| data_model_doc | string | 否 | "docs/design/data-model.md" |
| database_design_doc | string | 否 | "docs/design/database-design.md" |
| review_depth | string | 否 | "standard" |

### 执行过程

#### 步骤1：需求解析
提取需求：新建商品库存表（inventory），包含商品ID和库存数量。

#### 步骤2：文档加载
加载现有设计文档，发现Product实体已包含stock字段。

#### 步骤3：实体映射
检测到冲突：
- 需求设计：新建inventory表
- 现有设计：Product已包含stock字段

#### 步骤4：影响分析
分析两种方案的影响：
- 方案A：扩展现有products表
- 方案B：新建inventory表，迁移数据

#### 步骤5：冲突检测
🔴 设计冲突：
- 实体重复：Product已包含stock字段
- 关系冲突：新建独立inventory表与现有设计不一致

#### 步骤6：可行性评估
两种方案均可行，但方案A更优（避免重复建设和数据迁移）。

#### 步骤7：风险识别
- 方案A风险：低
- 方案B风险：中（数据迁移、重复建设）

#### 步骤8：报告生成
生成评审报告，提出解决方案。

### 最终输出
```markdown
# 需求评审报告

## 评审概要
- 需求名称: 商品库存管理
- 评审时间: 2024-01-15 17:00:00
- 评审深度: standard
- 评审结论: ❌ 不通过（存在设计冲突）

## 冲突检测结果

### 🔴 设计冲突
| 冲突类型 | 现有设计 | 需求设计 | 建议 |
|----------|----------|----------|------|
| 实体重复 | Product已包含stock字段 | 新建inventory表 | 扩展现有Product实体 |
| 关系冲突 | Product与Inventory一对一 | 新建独立inventory表 | 建议复用现有设计 |

### 现有设计分析
```sql
-- 现有商品表已包含库存字段
CREATE TABLE products (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    name VARCHAR(200) NOT NULL,
    stock INT DEFAULT 0,  -- 已存在库存字段
    ...
);
```

### 建议方案
