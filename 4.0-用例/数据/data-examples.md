## 用例1：订单数据质量检查

### 场景描述
数据工程师需要对订单表进行全面的数据质量检查，发现数据异常并提供修复建议。

### 使用Skill
- **Skill**: data-quality-check
- **触发方式**: `/data-quality-check` 或自然语言 "检查数据质量"

### 完整提示词
```
/data-quality-check

数据源: orders_table
检查类型: 全部
阈值: 80分
```

### 输入参数
| 参数名 | 类型 | 必填 | 值 |

## 问题详情

### 🔴 严重问题

#### 1. 跨表数据不一致
| 详情 | 内容 |
|------|------|
| 问题描述 | order.user_id 存在指向不存在用户的记录 |
| 影响范围 | 234条订单记录 |
| 影响 | 报表统计不准确，用户查询失败 |
| 建议 | 清理孤立订单，添加外键约束 |

```sql
-- 查找孤立订单
SELECT o.id, o.order_no, o.user_id
FROM orders o
LEFT JOIN users u ON o.user_id = u.id
WHERE u.id IS NULL;

-- 修复建议
DELETE o FROM orders o
LEFT JOIN users u ON o.user_id = u.id
WHERE u.id IS NULL;

-- 添加外键约束
ALTER TABLE orders
ADD CONSTRAINT fk_user_id
FOREIGN KEY (user_id) REFERENCES users(id);
```

#### 2. 时间戳异常
| 详情 | 内容 |
|------|------|
| 问题描述 | created_at 字段存在未来时间 |
| 影响范围 | 12条记录 |
| 影响 | 时间序列分析错误，报表异常 |
| 建议 | 修正错误数据，检查时区配置 |

```sql
-- 查找异常时间记录
SELECT id, order_no, created_at
FROM orders
WHERE created_at > NOW();

-- 修正数据
UPDATE orders
SET created_at = DATE_SUB(created_at, INTERVAL 100 YEAR)
WHERE created_at > '2099-01-01';
```

### 🟡 中等问题

#### 3. 字段缺失
| 字段 | 缺失数 | 占比 | 影响 |
|------|--------|------|------|

| remark | 89,000 | 7% | 无影响（可选字段） |
| coupon_id | 234,000 | 19% | 正常（无优惠订单） |

**分析**: delivery_address 缺失可能是因为虚拟商品订单，需要确认业务场景。

#### 4. 重复订单
| 类型 | 数量 | 说明 |
|------|------|------|
| 完全重复 | 0 | 无问题 |
| order_no重复 | 156 | 同一订单号多条记录 |

```sql
-- 查找重复订单号
SELECT order_no, COUNT(*) as cnt
FROM orders
GROUP BY order_no
HAVING COUNT(*) > 1;
```

### 🟢 轻微问题

#### 5. 枚举值异常

|------|--------|------|------|
| status | 'UNKNOWN' | 45 | 改为 'pending' |
| status | 'CANCEL' | 12 | 改为 'cancelled' |

## 用例2：数据血缘追踪

### 场景描述
数据架构师需要追踪订单明细宽表的数据血缘关系，了解数据来源和下游影响，以便进行变更影响分析。

### 使用Skill
- **Skill**: data-lineage-trace
- **触发方式**: `/data-lineage-trace` 或自然语言 "追踪数据血缘"

### 完整提示词
```
/data-lineage-trace

目标表: dwd_order_detail
方向: both (上游+下游)
深度: 3层
```

### 输入参数
| 参数名 | 类型 | 必填 | 值 |
|--------|------|------|-----|
| target | string | 是 | "dwd_order_detail" |
| direction | string | 否 | "both" |
| depth | number | 否 | 3 |

### 执行过程

#### 步骤1：目标识别
解析追踪目标：

**目标信息**：
- 目标表: dwd_order_detail (订单明细宽表)
- 表类型: DWD层（明细数据层）
- 核心字段: order_id, user_id, product_id, amount, user_name, product_name
- 数据量: 1.2亿条记录

#### 步骤2：血缘解析
解析SQL/代码获取血缘关系：

**解析范围**：
- ETL脚本: 15个
- SQL文件: 23个
- 配置文件: 8个

**直接依赖识别**：
- ods_orders (原始订单表) - 核心来源
- ods_users (原始用户表) - 关联维度
- dim_product (商品维度表) - 关联维度

#### 步骤3：关系构建
构建血缘关系图：

**上游依赖链**：
```
MySQL业务库 → ods_orders → dwd_order_detail
MySQL业务库 → ods_users → dwd_order_detail
MongoDB商品库 → dim_product → dwd_order_detail

**下游消费链**：
```
dwd_order_detail → ads_sales → BI报表系统
dwd_order_detail → ads_user → 用户分析服务
dwd_order_detail → rpt_dashboard → 运营看板
dwd_order_detail → 推荐服务/风控系统
```

#### 步骤4：可视化输出
生成血缘图谱：

**血缘图谱结构**：
- 第3层上游: MySQL业务库、MongoDB、API接口
- 第2层上游: ODS层表（ods_orders, ods_users）
- 第1层上游: DWD层表（dim_product）
- 目标表: dwd_order_detail
- 第1层下游: ADS层表（ads_sales, ads_user）
- 第2层下游: 应用系统（BI报表、推荐服务、风控系统）

#### 步骤5：影响分析
分析变更影响范围：

**影响统计**：
- 上游依赖: 6个数据源
- 下游影响: 8个系统/服务
- 风险等级: 🔴 高

**变更影响评估**：
- 新增字段: ✅ 低风险，下游兼容
- 删除字段: ❌ 高风险，需通知下游8个系统
- 修改字段: ⚠️ 中风险，需评估兼容性
