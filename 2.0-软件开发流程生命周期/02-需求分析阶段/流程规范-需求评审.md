# 流程规范：需求评审

## 基本信息

| 属性 | 值 |
|------|-----|
| 流程名称 | 需求评审 |
| 所属阶段 | 需求分析阶段 |
| 流程负责人 | 产品经理 |
| 前置流程 | 需求分析 |
| 后置流程 | 设计阶段 |

## 流程目的

确保需求文档质量，获得干系人对需求的认可。

## 触发条件

- PRD文档已完成
- 用户故事已编写

## 输入物

| 输入物 | 来源 | 必要性 |
|--------|------|--------|
| 产品需求文档(PRD) | 需求分析 | 必须 |
| 用户故事列表 | 需求分析 | 必须 |

## 执行步骤

### Step 1: 评审准备

- [ ] 发送评审邀请
- [ ] 分发评审材料
- [ ] 预约评审会议室

### Step 2: 评审会议

- [ ] 产品经理讲解需求
- [ ] 干系人提问和讨论
- [ ] 记录评审意见
- [ ] 确认修改项

### Step 3: 修改完善

- [ ] 根据评审意见修改文档
- [ ] 更新用户故事和验收标准
- [ ] 通知评审结果

### Step 4: 评审确认

- [ ] 获取干系人签字确认
- [ ] 归档评审记录

## 输出物

| 输出物 | 格式 | 存放位置 |
|--------|------|---------|
| 评审会议纪要 | 文档 | 项目文档库 |
| 评审确认书 | 文档 | 项目文档库 |
| 修订后的PRD | 文档 | 项目文档库 |

## 质量标准

- 关键干系人出席率 100%
- 评审意见已全部处理
- 获得干系人签字确认

## 相关Skill

| Skill | 用途 |
|-------|------|
| gate-check | 需求阶段门控检查 |

## 异常处理

| 异常场景 | 处理方式 |
|---------|---------|
| 评审不通过 | 根据意见修改后重新评审 |
| 干系人缺席 | 另行安排补充评审 |
