# 流程规范：设计评审

## 基本信息

| 属性 | 值 |
|------|-----|
| 流程名称 | 设计评审 |
| 所属阶段 | 设计阶段 |
| 流程负责人 | 技术负责人 |
| 前置流程 | 各设计流程完成 |
| 后置流程 | 开发阶段 |

## 流程目的

确保设计文档质量，发现潜在问题，获得团队对设计的认可。

## 触发条件

- 架构设计完成
- 数据库设计完成
- API设计完成
- UI/UX设计完成

## 输入物

| 输入物 | 来源 | 必要性 |
|--------|------|--------|
| 架构设计文档 | 架构设计 | 必须 |
| 数据库设计文档 | 数据库设计 | 必须 |
| API设计文档 | API设计 | 必须 |
| UI原型 | UI/UX设计 | 必须 |

## 执行步骤

### Step 1: 评审准备

- [ ] 收集评审材料
- [ ] 发送评审邀请
- [ ] 预约评审时间

### Step 2: 评审会议

- [ ] 技术负责人讲解架构
- [ ] 研发讲解数据库和API设计
- [ ] 设计师讲解UI设计
- [ ] 记录评审意见

### Step 3: 问题处理

- [ ] 整理评审问题清单
- [ ] 分配修改责任人
- [ ] 设定修改期限

### Step 4: 修改确认

- [ ] 完成修改
- [ ] 确认问题已解决
- [ ] 获取签字确认

## 输出物

| 输出物 | 格式 | 存放位置 |
|--------|------|---------|
| 评审会议纪要 | 文档 | 项目文档库 |
| 问题清单 | 表格 | 项目文档库 |
| 设计确认书 | 文档 | 项目文档库 |

## 质量标准

- 关键人员出席率 100%
- 评审问题已全部处理
- 获得签字确认

## 相关Skill

| Skill | 用途 |
|-------|------|
| gate-check | 设计阶段门控检查 |

## 异常处理

| 异常场景 | 处理方式 |
|---------|---------|
| 评审不通过 | 修改后重新评审 |
| 重大设计缺陷 | 重新设计方案 |
