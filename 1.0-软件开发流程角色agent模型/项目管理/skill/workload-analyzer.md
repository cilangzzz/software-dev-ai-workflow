## 触发条件

| 触发方式 | 条件 |
|---------|------|
| 关键词 | "负载"、"工时分配"、"人员缺口"、"负荷率" |
| 命令 | `/workload` |
| 事件 | 排期设计阶段、人员变更 |

## 输入参数

| 参数名 | 类型 | 必填 | 默认值 | 描述 |
|-------|------|------|--------|------|
| task_allocation | array | true | - | 任务分配列表（任务ID、负责人、工时） |
| team_members | array | true | - | 团队成员列表（姓名、角色、可用工时） |
| sprint_duration | number | false | 14 | Sprint周期（天） |

## 执行流程

### Phase 1: 可用工时计算

| 计算项 | 公式 | 说明 |
|-------|------|------|
| 总工作日 | Sprint周期天数 | 默认14天 |
| 有效工作日 | 总工作日 - 休假/培训 | 实际可用天数 |
| 可用工时 | 有效工作日 × 有效系数(0.8) | 考虑会议、沟通时间 |

### Phase 2: 分配工时统计

| 统计项 | 方法 | 输出 |
|-------|------|------|
| 总分配工时 | 按人员汇总分配工时 | 各人员分配值 |
| 高优先级工时 | 高优任务工时汇总 | 高优工时占比 |
| 低优先级工时 | 低优任务工时汇总 | 可调整空间 |

### Phase 3: 负载率计算

| 指标 | 公式 | 预警阈值 |
|------|------|---------|
| 负载率 | 分配工时/可用工时 | >70%预警, >90%严重 |
| 高优负载率 | 高优工时/可用工时 | >50%预警 |
| 缺口工时 | 分配工时-可用工时 | >0需协调 |

### Phase 4: 缺口识别

| 缺口类型 | 识别条件 | 处理建议 |
|---------|---------|---------|
| 人员缺口 | 任务未分配负责人 | 协调人员或外包 |
| 工时缺口 | 负载率>90% | 延期低优先级或协调额外人员 |
| 技能缺口 | 技术栈不匹配 | 培训或协调匹配人员 |

### 缺口处理建议

| 场景 | 建议方案 |
|------|---------|
| 负载率<50% | 可增加任务，提高利用率 |
| 负载率50-70% | 正常，可接受低优先级任务 |
| 负载率70-90% | 预警，避免增加新任务 |
| 负载率>90% | 严重，需要调整或协调 |

## 注意事项

1. **有效系数**: 可用工时按80%计算，预留会议、沟通时间
2. **预警阈值**: 负载率超过70%需要预警，超过90%必须调整
3. **关键路径**: 关键路径人员不能超负载
4. **缓冲空间**: 低优先级任务作为缓冲，可延期或删除
