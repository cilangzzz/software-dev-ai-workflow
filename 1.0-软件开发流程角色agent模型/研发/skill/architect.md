# Skill: architect

## 基本信息
- **名称**: architect
- **版本**: 1.0.0
- **所属部门**: 研发部
- **优先级**: P0

## 功能描述
架构设计主入口，根据需求自动生成架构方案和架构决策记录（ADR）。提供技术选型建议、系统架构图、模块划分等核心架构输出。

## 触发条件
- 命令触发: `/architect`
- 自然语言触发:
  - "设计系统架构"
  - "生成架构方案"
  - "技术选型建议"
  - "架构设计"

## 输入参数
| 参数名 | 类型 | 必填 | 描述 |
|--------|------|------|------|
| requirements | string | 是 | 需求描述或PRD内容 |
| constraints | string | 否 | 技术约束条件 |
| preferences | string | 否 | 技术偏好（语言、框架等） |
| scale | string | 否 | 系统规模：small / medium / large |

## 执行流程
1. **需求分析** - 解析需求中的功能和非功能需求
2. **技术选型** - 推荐技术栈方案
3. **架构设计** - 生成系统架构图和模块划分
4. **决策记录** - 自动生成ADR文档
5. **风险评估** - 识别架构风险点
6. **方案对比** - 提供备选方案及权衡分析

## 质量标准
- 技术选型合理性：提供至少2个备选方案
- 架构完整性：覆盖所有功能需求
- 可扩展性：支持3-5倍业务增长
- 文档完整性：包含架构图、模块图、接口定义

## 使用示例

### 示例1：电商系统架构设计
**输入**:
```
需求：开发一个B2C电商平台，支持用户注册登录、商品浏览、购物车、订单管理、
支付功能。预计日均PV 100万，高峰期QPS 5000。
约束：团队熟悉Java和MySQL，预算有限。
```

**输出**:
```markdown
# 电商系统架构设计

## 1. 架构概述
采用微服务架构，前后端分离，支持水平扩展。核心服务包括用户服务、商品服务、
订单服务、支付服务等。

## 2. 技术选型
| 层级 | 技术选择 | 选择理由 |
|------|---------|---------|
| 前端 | Vue 3 + Vite | 团队熟悉，生态完善，性能优秀 |
| 后端 | Spring Boot 3 | Java生态，团队熟悉，成熟稳定 |
| 数据库 | MySQL 8.0 | 关系型数据库，团队熟悉，成本低 |
| 缓存 | Redis | 高性能缓存，支持分布式 |
| 消息队列 | RabbitMQ | 轻量级，成本低，满足当前需求 |
| 网关 | Spring Cloud Gateway | 统一入口，限流熔断 |

## 3. 系统架构图
```
                    ┌─────────────┐
                    │   CDN/Nginx │
                    └──────┬──────┘
                           │
                    ┌──────▼──────┐
                    │  API Gateway│
                    └──────┬──────┘
                           │
        ┌──────────────────┼──────────────────┐
        │                  │                  │
┌───────▼───────┐ ┌───────▼───────┐ ┌───────▼───────┐
│  User Service │ │ Goods Service │ │ Order Service │
└───────┬───────┘ └───────┬───────┘ └───────┬───────┘
        │                  │                  │
        └──────────────────┼──────────────────┘
                           │
                    ┌──────▼──────┐
                    │    MySQL    │
                    │   (主从)    │
                    └─────────────┘
```

## 4. 模块划分
### 4.1 核心服务
- **用户服务(user-service)**: 用户注册、登录、权限管理
- **商品服务(goods-service)**: 商品管理、分类、搜索
- **订单服务(order-service)**: 订单创建、状态管理
- **支付服务(payment-service)**: 支付对接、退款

### 4.2 公共模块
- **common-core**: 公共工具类、异常处理
- **common-redis**: Redis配置和工具
- **common-security**: 安全认证组件

## 5. 风险点
| 风险 | 影响 | 缓解措施 |
|------|------|---------|
| 单体数据库瓶颈 | 高并发时性能下降 | 分库分表、读写分离 |
| 服务雪崩 | 系统不可用 | 熔断降级、限流 |
| 数据一致性 | 订单状态异常 | 分布式事务、最终一致性 |

## 6. 相关ADR
- ADR-001: 选择Spring Boot作为后端框架
- ADR-002: 使用Redis作为分布式缓存
- ADR-003: 采用最终一致性模型
```

## 依赖工具
- Read - 读取需求文档

- Bash - 执行架构验证命令

## 注意事项
- 架构设计需要平衡理想方案和团队能力
- 过早优化是万恶之源，优先保证MVP可用
- 架构文档需要持续更新维护
- 重大架构决策必须记录ADR
- 考虑成本约束和运维复杂度
