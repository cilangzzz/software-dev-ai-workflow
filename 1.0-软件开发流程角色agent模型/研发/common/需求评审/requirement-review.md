# Skill: requirement-review

## 基本信息
- **名称**: requirement-review
- **版本**: 1.0.0
- **所属部门**: 研发部
- **优先级**: P0

## 功能描述
辅助研发团队进行需求评审，结合现有数据模型、领域设计、数据库设计文档，自动分析需求完整性、识别设计冲突、评估变更影响、生成评审报告。帮助研发在需求评审阶段提前发现潜在问题，降低后续开发风险。

## 触发条件
- 命令触发: `/requirement-review`
- 自然语言触发:
  - "评审需求"
  - "需求评审"
  - "分析这个需求的影响"
  - "检查需求与现有设计的兼容性"
  - "评估需求可行性"

## 输入参数
| 参数名 | 类型 | 必填 | 描述 |
|--------|------|------|------|
| requirement_doc | string | 是 | 需求文档路径或内容（PRD/用户故事） |
| data_model_doc | string | 否 | 数据模型文档路径 |
| domain_design_doc | string | 否 | 领域设计文档路径 |
| database_design_doc | string | 否 | 数据库设计文档路径 |
| review_depth | string | 否 | 评审深度：quick(快速) / standard(标准) / deep(深度) |
| focus_areas | string | 否 | 关注领域：data(数据) / api(接口) / performance(性能) / all |

## 执行流程
1. **需求解析** - 解析需求文档，提取功能点、实体、业务规则
2. **文档加载** - 加载现有设计文档（数据模型、领域设计、数据库设计）
3. **实体映射** - 将需求中的实体与现有数据模型进行匹配
4. **影响分析** - 分析需求对现有设计的影响范围
5. **冲突检测** - 识别需求与现有设计的潜在冲突
6. **可行性评估** - 评估技术可行性和实现复杂度
7. **风险识别** - 识别技术风险和业务风险
8. **报告生成** - 生成需求评审报告

## 使用示例

### 示例1：电商订单退款功能评审
**输入**:
```
requirement_doc: docs/requirements/order-refund.md
data_model_doc: docs/design/data-model.md
domain_design_doc: docs/design/domain-design.md
database_design_doc: docs/design/database-design.md
review_depth: standard
```

**输出**:
```markdown
# 需求评审报告

## 评审概要
- 需求名称: 订单退款功能
- 评审时间: 2024-01-15 10:30:00
- 评审深度: standard
- 评审结论: ⚠️ 有条件通过

## 需求摘要
实现订单退款功能，支持用户在订单完成后申请退款，包括：
1. 用户申请退款（全额/部分退款）
2. 商家审核退款申请
3. 退款进度查询
4. 退款到账通知

## 实体影响分析

### 新增实体
| 实体名称 | 来源字段 | 建议的数据模型位置 |
|----------|----------|-------------------|
| RefundOrder | orderNo, refundAmount, reason, status | 订单域-退款聚合 |

### 修改实体
| 实体名称 | 变更类型 | 新增属性 | 影响范围 |
|----------|----------|----------|----------|
| Order | 属性扩展 | refundStatus, refundAmount | 订单查询、订单列表 |

### 复用实体
| 实体名称 | 复用方式 | 现有属性满足度 |
|----------|----------|---------------|
| User | 关联引用 | ✅ 完全满足 |
| Payment | 关联引用 | ⚠️ 部分满足（需新增退款流水记录） |

## 数据库影响分析

### 表结构变更
| 变更类型 | 表名 | 变更详情 | 影响等级 |
|----------|------|----------|----------|
| 新增 | refund_order | 退款订单主表 | 🟠 中 |
| 新增 | refund_item | 退款商品明细表 | 🟠 中 |
| 修改 | orders | 新增refund_status, refund_amount字段 | 🔴 高 |
| 修改 | payment_record | 新增refund_trace_id字段 | 🟢 低 |

### 数据迁移需求
| 迁移场景 | 涉及表 | 数据量预估 | 复杂度 |
|----------|--------|-----------|--------|
| 历史订单退款状态初始化 | orders | 500万+ | 中 |

```sql
-- 建议的迁移脚本
ALTER TABLE orders
ADD COLUMN refund_status TINYINT DEFAULT 0 COMMENT '退款状态: 0-无退款, 1-退款中, 2-已退款',
ADD COLUMN refund_amount DECIMAL(10,2) DEFAULT 0.00 COMMENT '已退款金额';

-- 历史数据处理
UPDATE orders SET refund_status = 0, refund_amount = 0
WHERE refund_status IS NULL;
```

## 风险识别

### 🔴 高风险
| 风险描述 | 影响范围 | 缓解措施建议 |
|----------|----------|--------------|
| 退款与支付回调并发 | 可能导致状态不一致 | 使用分布式锁+状态机 |
| 大额退款资金安全 | 财务风险 | 增加风控审核环节 |

### 🟠 中风险
| 风险描述 | 影响范围 | 缓解措施建议 |
|----------|----------|--------------|
| 部分退款库存回滚 | 库存数据准确性 | 明确部分退款库存处理规则 |
| 历史订单表结构变更 | 500万+数据迁移 | 分批迁移，灰度验证 |

## 待澄清问题

| # | 问题 | 优先级 | 提问对象 | 状态 |
|---|------|--------|----------|------|
| 1 | 部分退款时，已使用的优惠券如何处理？ | P0 | 产品 | 待确认 |
| 2 | 退款审核是否需要多级审核？ | P0 | 产品/业务 | 待确认 |
| 3 | 退款超时未审核如何处理？ | P1 | 产品 | 待确认 |

## 评审结论

### 总体评价
需求清晰，技术方案可行。主要风险在于退款与支付的并发处理、大额退款的资金安全。建议采用状态机模式管理退款流程，增加风控审核环节。

### 前置条件
1. 确认部分退款时优惠券处理规则
2. 确认退款审核流程（是否需要多级审核）
3. 制定大额退款风控方案

### 下一步
1. 产品确认待澄清问题后，进入技术方案设计阶段
2. 调用 `/architect` 生成退款模块详细技术方案
3. 调用 `/adr` 记录退款流程关键决策
```
