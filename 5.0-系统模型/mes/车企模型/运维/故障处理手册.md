## 一、故障分类与分级

### 1.1 故障级别定义

| 级别 | 名称 | 定义 | 响应时间 | 处理时间 |
|------|------|------|---------|---------|
| P0 | 紧急故障 | 系统完全不可用，影响生产运行 | 15分钟 | 2小时 |
| P1 | 严重故障 | 核心功能不可用，严重影响业务 | 30分钟 | 4小时 |
| P2 | 一般故障 | 次要功能异常，部分影响业务 | 2小时 | 8小时 |
| P3 | 轻微故障 | 不影响核心业务，体验问题 | 4小时 | 24小时 |

### 1.2 故障类型分类

| 类型 | 说明 | 典型场景 |
|------|------|---------|
| 服务故障 | 应用服务无法正常提供服务 | 服务宕机、进程异常退出 |
| 数据库故障 | 数据库连接异常或性能问题 | 连接池耗尽、死锁、慢查询 |
| 缓存故障 | Redis服务异常 | 连接超时、内存溢出 |
| 网络故障 | 网络连通性问题 | 连接超时、DNS解析失败 |
| 设备故障 | 生产设备集成问题 | 扭矩枪离线、PLC通讯中断 |
| 性能故障 | 系统响应缓慢 | CPU/内存过高、IO瓶颈 |

### 2.2 常用排查命令

#### 服务状态检查
```bash
# 查看服务进程
ps -ef | grep yudao-server

# 查看端口占用
netstat -tlnp | grep 8080

# 查看服务健康状态
curl http://localhost:8080/actuator/health

# 查看JVM状态
jstat -gc <pid>
jstack <pid> > thread_dump.txt
jmap -heap <pid>
```

#### 日志查看
```bash
# 查看应用日志
tail -f /opt/mes/logs/mes-server.log

# 查看错误日志
tail -f /opt/mes/logs/mes-error.log

# 搜索特定异常
grep -i "exception" /opt/mes/logs/mes-error.log | tail -50

# 查看Nginx日志
tail -f /var/log/nginx/error.log
```

#### 数据库检查
```bash
# 检查MySQL连接
mysql -u root -p -e "SHOW PROCESSLIST;"

# 检查慢查询
mysql -u root -p -e "SELECT * FROM mysql.slow_log ORDER BY query_time DESC LIMIT 10;"

# 检查锁等待
mysql -u root -p -e "SELECT * FROM information_schema.INNODB_LOCK_WAITS;"
```

#### Redis检查
```bash
# 检查Redis状态
redis-cli ping

# 查看Redis信息
redis-cli info

# 查看内存使用
redis-cli info memory | grep used_memory_human
```

## 三、常见故障处理

### 3.1 应用服务故障

#### 故障：服务无法启动

**现象**：应用服务启动失败，进程不存在

**排查步骤**：
```bash
# 1. 检查日志
tail -n 100 /opt/mes/logs/mes-server.log

# 2. 检查端口是否被占用
netstat -tlnp | grep 8080

# 3. 检查配置文件
cat /opt/mes/config/application.yml

# 4. 检查Java环境
java -version
```

**常见原因及解决方案**：

| 原因 | 解决方案 |
|------|---------|
| 端口被占用 | `kill -9 <pid>` 或修改端口配置 |
| 配置文件错误 | 检查YAML语法，修正配置项 |
| 依赖服务不可用 | 先启动MySQL/Redis等依赖服务 |
| 内存不足 | 调整JVM参数 `-Xmx` |
| Java版本不匹配 | 安装正确的JDK版本 |

#### 故障：服务响应超时

**现象**：接口响应缓慢或超时

**排查步骤**：
```bash
# 1. 检查系统资源
top -p <pid>

# 2. 查看线程状态
jstack <pid> | grep -A 10 "BLOCKED"

# 3. 检查GC情况
jstat -gc <pid> 1000 10

# 4. 检查数据库连接
mysql -u root -p -e "SHOW PROCESSLIST;"
```

**解决方案**：
1. CPU/内存过高：排查是否存在死循环或内存泄漏
2. GC频繁：调整堆内存大小，优化对象创建
3. 数据库连接耗尽：增大连接池或排查慢查询
4. 线程阻塞：分析线程堆栈，解决锁竞争

#### 故障：服务OOM崩溃

**现象**：服务因内存溢出自动退出

**排查步骤**：
```bash
# 1. 查看堆转储文件
ls -la /data/heapdump/

# 2. 分析堆转储（使用MAT或jhat）
jhat heapdump.hprof

# 3. 检查JVM配置
ps -ef | grep yudao-server
```

**解决方案**：
1. 增大堆内存：`-Xmx4g -Xms4g`
2. 优化代码，减少大对象创建
3. 增加缓存过期策略
4. 排查内存泄漏

---

### 3.2 数据库故障

#### 故障：数据库连接失败

**现象**：应用无法连接数据库

**排查步骤**：
```bash
# 1. 检查MySQL服务状态
systemctl status mysqld

# 2. 测试数据库连接
mysql -h <host> -u <user> -p

# 3. 检查网络连通性
telnet <db_host> 3306

# 4. 检查用户权限
mysql -u root -p -e "SELECT Host, User FROM mysql.user;"
```

**解决方案**：
| 原因 | 解决方案 |
|------|---------|
| MySQL服务未启动 | `systemctl start mysqld` |
| 防火墙阻止 | 开放3306端口 |
| 用户权限不足 | `GRANT ALL ON mes_prod.* TO 'user'@'%';` |
| 连接数超限 | 调整 `max_connections` 参数 |

#### 故障：数据库死锁

**现象**：事务长时间等待，业务操作卡住

**排查步骤**：
```sql
-- 查看当前锁
SELECT * FROM information_schema.INNODB_LOCKS;
SELECT * FROM information_schema.INNODB_LOCK_WAITS;

-- 查看当前事务
SELECT * FROM information_schema.INNODB_TRX;
```

**解决方案**：
```sql
-- 杀掉阻塞的事务
KILL <trx_id>;

-- 优化SQL，减少锁范围
-- 调整事务隔离级别
```

#### 故障：慢查询影响性能

**现象**：数据库CPU高，查询响应慢

**排查步骤**：
```sql
-- 查看慢查询日志
SELECT * FROM mysql.slow_log ORDER BY query_time DESC LIMIT 10;

-- 分析执行计划
EXPLAIN SELECT ...;

-- 查看表索引
SHOW INDEX FROM <table>;
```

**解决方案**：
1. 添加合适的索引
2. 优化SQL语句
3. 分表分库
4. 引入缓存

---

### 3.3 缓存故障

#### 故障：Redis连接失败

**现象**：应用无法连接Redis

**排查步骤**：
```bash
# 1. 检查Redis服务
systemctl status redis

# 2. 测试连接
redis-cli ping

# 3. 检查网络
telnet <redis_host> 6379
```

**解决方案**：
| 原因 | 解决方案 |
|------|---------|
| 服务未启动 | `systemctl start redis` |
| 密码错误 | 检查配置中的密码 |
| 网络不通 | 检查防火墙规则 |
| 连接数超限 | 调整 `maxclients` |

#### 故障：Redis内存溢出

**现象**：Redis内存使用率过高，写入失败

**排查步骤**：
```bash
# 查看内存使用
redis-cli info memory

# 查看大Key
redis-cli --bigkeys

# 查看Key分布
redis-cli --scan --pattern "*" | head -100
```

**解决方案**：
1. 设置过期时间：`EXPIRE key seconds`
2. 配置内存淘汰策略：`maxmemory-policy allkeys-lru`
3. 清理无用Key
4. 增加Redis内存

### 3.4 设备集成故障

#### 故障：扭矩枪数据不上传

**现象**：扭矩枪拧紧数据未同步到系统

**排查步骤**：
```bash
# 1. 检查设备网络
ping <torque_gun_ip>

# 2. 检查MQTT连接
redis-cli
> PUBSUB CHANNELS device.*

# 3. 查看IoT服务日志
tail -f /opt/mes/logs/iot-service.log

# 4. 检查消息队列
rabbitmqctl list_queues | grep device
```

**解决方案**：
| 原因 | 解决方案 |
|------|---------|
| 设备离线 | 检查设备电源和网络 |
| MQTT连接断开 | 重启设备或IoT网关服务 |
| 消息队列积压 | 清理队列或增加消费者 |
| 数据格式错误 | 检查协议配置 |

#### 故障：扫描枪无法扫码

**现象**：终端扫码无反应或报错

**排查步骤**：
1. 检查扫描枪驱动是否正常
2. 检查终端应用是否响应
3. 查看终端日志错误信息
4. 测试扫描枪在其他应用中是否正常

**解决方案**：
1. 重启扫描枪或重新配对
2. 更新终端应用版本
3. 检查USB/蓝牙连接
4. 更换扫描枪设备

---

### 3.5 性能故障

#### 故障：系统响应缓慢

**现象**：页面加载慢，接口响应超时

**排查步骤**：
```bash
# 1. 检查服务器资源
top
free -m
df -h

# 2. 检查应用状态
curl http://localhost:8080/actuator/metrics

# 3. 检查数据库
mysql -u root -p -e "SHOW PROCESSLIST;"

# 4. 检查网络
ping <server_ip>
traceroute <server_ip>
```

**解决方案**：
1. CPU高：排查CPU密集型任务
2. 内存高：重启服务或调整JVM
3. IO高：检查磁盘读写，优化SQL
4. 网络延迟：检查网络设备

## 四、应急预案

### 4.1 服务熔断

当服务出现异常时，可通过以下方式保护系统：

```bash
# 1. 限流（Nginx配置）
# 在http块添加
limit_req_zone $binary_remote_addr zone=api_limit:10m rate=10r/s;

# 2. 降级（关闭非核心功能）
# 在配置文件中设置
feature:
  dashboard: false
  report: false

# 3. 熔断（Hystrix配置）
hystrix:
  command:
    default:
      circuitBreaker:
        enabled: true
        requestVolumeThreshold: 20
```

### 4.2 快速恢复

```bash
# 1. 重启应用服务
./bin/restart.sh

# 2. 清理Redis缓存
redis-cli FLUSHDB

# 3. 重启依赖服务
systemctl restart mysqld
systemctl restart redis
systemctl restart rabbitmq-server

# 4. 回滚到上一版本
cp /data/releases/yudao-server-last.jar yudao-server.jar
./bin/restart.sh
```

### 4.3 数据恢复

```bash
# 恢复最近的数据库备份
gunzip /data/backup/mysql/mes_prod_latest.sql.gz
mysql -u root -p mes_prod < mes_prod_latest.sql
```
