# Skill: user-story-generator

## 基本信息
- **名称**: user-story-generator
- **版本**: 1.0.0
- **所属部门**: 产品部
- **优先级**: P0

## 功能描述
将需求描述或PRD内容转化为符合INVEST原则的标准用户故事格式。自动识别用户角色、行为和目标，生成可执行的用户故事卡片。

## 触发条件
- 命令触发: `/user-story-generator`
- 自然语言触发:
  - "生成用户故事"
  - "转换为用户故事格式"
  - "帮我写用户故事"
  - "需求转story"

## 输入参数
| 参数名 | 类型 | 必填 | 描述 |
|--------|------|------|------|
| requirement_content | string | 是 | 需求描述或PRD内容 |
| story_format | string | 否 | 故事格式：standard(标准) / bdd(BDD格式) |
| detail_level | string | 否 | 详细程度：brief(简要) / detailed(详细) |

## 执行流程
1. **角色识别** - 从需求中识别用户角色
2. **行为提取** - 提取用户的行为动作
3. **目标推导** - 推导用户行为背后的目标
4. **INVEST检验** - 检验故事是否符合INVEST原则
5. **拆分建议** - 对过大的故事提供拆分建议
6. **格式输出** - 按指定格式输出用户故事

## INVEST原则
- **I**ndependent（独立的）- 故事之间不应有强依赖
- **N**egotiable（可协商的）- 故事细节可讨论
- **V**aluable（有价值的）- 对用户或业务有价值
- **E**stimable（可估算的）- 开发可以估算工作量
- **S**mall（足够小的）- 可以在一个迭代内完成
- **T**estable（可测试的）- 可以编写测试用例

## 输出格式

### 标准格式
```markdown
# 用户故事

## 故事卡片
**作为** [用户角色]
**我想要** [行为/功能]
**以便于** [目标/价值]

## 验收标准
- Given [前置条件]
- When [触发动作]
- Then [预期结果]

## 故事详情
- 优先级: P0/P1/P2
- 故事点: [估算值]
- Epic: [所属Epic]
- Sprint: [目标迭代]

## 子任务
- [ ] 任务1
- [ ] 任务2

## 注意事项
- [技术或业务注意事项]
```

## 质量标准
- INVEST原则符合率 ≥ 90%
- 用户角色识别准确率 ≥ 95%
- 故事粒度合理（建议1-8故事点范围）
- 验收标准可测试性 100%

## 使用示例

### 示例1：功能需求转用户故事
**输入**:
```
需求：用户可以在个人中心查看自己的历史订单，包括订单编号、下单时间、
商品信息、订单金额和订单状态。支持按时间筛选和订单状态筛选。
```

**输出**:
```markdown
# 用户故事：历史订单查询

## 故事卡片
**作为** 注册用户
**我想要** 在个人中心查看历史订单列表
**以便于** 追踪我的购物记录和订单状态

## 验收标准

### 场景1：查看订单列表
- Given 用户已登录并进入个人中心
- When 用户点击"我的订单"
- Then 显示历史订单列表，包含订单编号、下单时间、商品信息、金额、状态

### 场景2：按时间筛选
- Given 用户在订单列表页
- When 用户选择时间范围筛选
- Then 订单列表按选定时间范围过滤

### 场景3：按状态筛选
- Given 用户在订单列表页
- When 用户选择订单状态筛选
- Then 订单列表按选定状态过滤

## 故事详情
- 优先级: P1
- 故事点: 5
- Epic: 订单管理
- Sprint: 待分配

## 子任务
- [ ] 订单列表页面开发
- [ ] 时间筛选组件
- [ ] 状态筛选组件
- [ ] 分页加载逻辑
- [ ] 空状态展示

## 注意事项
- 订单数据需要分页加载（每页20条）
- 需要处理订单图片加载失败情况
- 移动端需要适配小屏幕
```

### 示例2：大故事拆分
**输入**:
```
需求：实现一个完整的购物车功能，用户可以添加商品、修改数量、删除商品、
选择优惠券、计算优惠价格、提交订单。
```

**输出**:
```markdown
# 用户故事拆分建议

原需求过大，建议拆分为以下用户故事：

## Story 1: 购物车基础管理
**作为** 购物用户
**我想要** 在购物车中添加、修改、删除商品
**以便于** 管理我想要购买的商品
- 故事点: 5
- 优先级: P0

## Story 2: 购物车价格计算
**作为** 购物用户
**我想要** 看到购物车商品的实时价格汇总
**以便于** 了解我需要支付的金额
- 故事点: 3
- 优先级: P0

## Story 3: 优惠券选择
**作为** 购物用户
**我想要** 在购物车选择可用优惠券
**以便于** 享受优惠价格
- 故事点: 5
- 优先级: P1

## Story 4: 提交订单
**作为** 购物用户
**我想要** 从购物车提交订单
**以便于** 完成购买流程
- 故事点: 8
- 优先级: P0
- 注意：建议进一步拆分
```

## 依赖工具
- Read - 读取PRD文档
- Write - 输出用户故事文档
- Grep - 搜索相关故事

## 注意事项
- 用户故事应该从用户视角出发，而非技术视角
- 验收标准要具体、可测试
- 过大的故事（>8故事点）需要拆分
- 注意识别跨故事的前置依赖关系
- 故事描述使用用户语言，避免技术术语
