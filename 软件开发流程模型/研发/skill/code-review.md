# Skill: code-review

## 基本信息
- **名称**: code-review
- **版本**: 1.0.0
- **所属部门**: 研发部
- **优先级**: P0

## 功能描述
对代码变更进行自动化审查，检测潜在问题并提供改进建议。覆盖代码质量、安全漏洞、性能问题、最佳实践等多个维度。

## 触发条件
- 命令触发: `/code-review`
- 自然语言触发:
  - "审查代码"
  - "检查代码质量"
  - "review代码"
  - "代码评审"

## 输入参数
| 参数名 | 类型 | 必填 | 描述 |
|--------|------|------|------|
| target | string | 是 | 审查目标：文件路径或Git diff |
| review_type | string | 否 | 审查类型：quick / full / security |
| focus | string | 否 | 关注点：quality / security / performance / all |

## 执行流程
1. **代码解析** - 解析目标代码或变更
2. **静态分析** - 检测代码异味和潜在问题
3. **安全扫描** - 检测安全漏洞
4. **性能检查** - 识别性能问题
5. **最佳实践** - 检查是否符合最佳实践
6. **报告生成** - 生成审查报告和改进建议

## 使用示例

### 示例1：审查用户服务代码
**输入**:
```
target: src/main/java/com/example/user/UserService.java
review_type: full
```

**输出**:
```markdown
# 代码审查报告

## 摘要
- 审查文件: 1个
- 代码行数: 150行
- 问题总数: 5个
  - 🔴 严重: 1个
  - 🟠 重要: 2个
  - 🟡 建议: 2个

## 问题列表

### 🔴 严重问题
| # | 文件 | 行号 | 问题描述 | 修复建议 |
|---|------|------|---------|---------|
| 1 | UserService.java | 45 | SQL注入风险：直接拼接用户输入 | 使用参数化查询或JPA方法命名 |

**详细代码**:
```java
// 问题代码 (第45行)
String sql = "SELECT * FROM users WHERE email = '" + email + "'";

// 修复建议
@Query("SELECT u FROM User u WHERE u.email = :email")
User findByEmail(@Param("email") String email);
```

### 🟠 重要问题
| # | 文件 | 行号 | 问题描述 | 修复建议 |
|---|------|------|---------|---------|
| 1 | UserService.java | 78 | 方法过长(45行)，影响可读性 | 拆分为多个私有方法 |
| 2 | UserService.java | 102 | 缺少事务注解，可能导致数据不一致 | 添加@Transactional注解 |

### 🟡 改进建议
| # | 文件 | 问题描述 | 建议 |
|---|------|---------|------|
| 1 | UserService.java | 魔法数字：超时时间硬编码为30000 | 提取为配置项 |
| 2 | UserService.java | 日志级别使用不当 | 关键操作使用info级别 |

## 代码质量评分
| 维度 | 评分 | 说明 |
|------|------|------|
| 可读性 | ⭐⭐⭐⭐ | 命名规范，结构清晰 |
| 可维护性 | ⭐⭐⭐ | 部分方法过长 |
| 安全性 | ⭐⭐⭐ | 存在SQL注入风险 |
| 性能 | ⭐⭐⭐⭐ | 无明显性能问题 |
| 测试覆盖 | ⭐⭐ | 缺少边界测试 |

## 详细分析

### 1. 安全问题
**SQL注入风险 (第45行)**
- 风险等级: 高
- 影响: 攻击者可窃取或篡改数据库数据
- 修复方案:
  1. 使用JPA Repository方法命名
  2. 或使用@Query注解参数化查询
  3. 切勿直接拼接用户输入

### 2. 性能问题
**潜在的N+1查询 (第89行)**
- 问题: 循环中调用findById可能产生N+1查询
- 建议: 使用JOIN FETCH一次性加载关联数据

## 总体评价
代码整体质量良好，命名规范清晰，但存在一个严重的安全问题需要立即修复。
建议：
1. 🔴 立即修复SQL注入问题
2. 🟠 重构过长的方法
3. 补充单元测试覆盖边界场景
```

## 检查规则

### 安全检查
- SQL注入
- XSS漏洞
- 敏感信息泄露
- 硬编码密码
- 不安全的随机数

### 性能检查
- N+1查询
- 内存泄漏
- 不必要的循环
- 过大的对象创建

### 代码质量
- 代码重复
- 过长方法
- 过深嵌套
- 魔法数字
- 缺少注释

## 依赖工具
- Read - 读取代码文件
- Grep - 搜索代码模式
- Bash - 执行静态分析工具

## 注意事项
- AI审查不能完全替代人工Code Review
- 严重问题必须修复后才能合并
- 建议性问题可根据实际情况决定是否修复
- 定期更新检查规则以适应新威胁
