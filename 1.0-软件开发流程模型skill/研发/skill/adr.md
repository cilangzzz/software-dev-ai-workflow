# Skill: adr

## 基本信息
- **名称**: adr
- **版本**: 1.0.0
- **所属部门**: 研发部
- **优先级**: P0

## 功能描述
创建和管理架构决策记录（Architecture Decision Record）。记录重要的技术决策及其背景、影响，建立可追溯的技术决策知识库。

## 触发条件
- 命令触发: `/adr`
- 自然语言触发:
  - "创建ADR"
  - "记录架构决策"
  - "写决策文档"
  - "技术决策记录"

## 输入参数
| 参数名 | 类型 | 必填 | 描述 |
|--------|------|------|------|
| title | string | 是 | 决策标题 |
| context | string | 是 | 决策背景和上下文 |
| decision | string | 否 | 决策内容（可选，AI可辅助生成） |
| alternatives | string | 否 | 备选方案列表 |

## 执行流程
1. **收集上下文** - 整理决策背景信息
2. **分析备选方案** - 列出可能的解决方案
3. **评估权衡** - 分析各方案的优劣
4. **形成决策** - 明确最终选择
5. **记录影响** - 预测决策的后果
6. **格式输出** - 生成标准ADR文档

## 质量标准
- 决策可追溯性：完整的上下文记录
- 权衡分析：至少列出2个备选方案
- 影响评估：包含正面和负面影响
- 简洁性：核心内容不超过2页

## 使用示例

### 示例1：数据库选型决策
**输入**:
```
title: 选择PostgreSQL作为主数据库
context: 新项目需要一个关系型数据库，需要支持JSON类型和全文搜索。
备选方案：MySQL, PostgreSQL, MariaDB
```

**输出**:
```markdown
# ADR-001: 选择PostgreSQL作为主数据库

## 状态
Accepted

## 背景
新开发的电商平台需要一个关系型数据库，主要需求包括：
1. 支持复杂查询和事务
2. 原生JSON支持（商品属性灵活存储）
3. 全文搜索能力（商品搜索）
4. 良好的性能和可扩展性
5. 团队有运维经验

## 决策
选择PostgreSQL作为主数据库。

理由：
1. 原生JSONB类型性能优于MySQL，适合商品灵活属性存储
2. 内置全文搜索功能，可减少Elasticsearch依赖
3. 更好的并发控制（MVCC实现）
4. 优秀的查询优化器
5. 开源免费，社区活跃

## 备选方案
### 方案1: MySQL 8.0
- 优点:
  - 团队更熟悉
  - 生态更成熟
  - 文档资料丰富
- 缺点:
  - JSON性能较弱
  - 全文搜索功能有限
- 不选择原因: JSON和搜索能力不满足需求

### 方案2: MariaDB
- 优点:
  - 完全兼容MySQL
  - 开源社区活跃
- 缺点:
  - 与MySQL类似的限制
  - 社区规模较小
- 不选择原因: 与MySQL相比无明显优势

## 影响
### 正面影响
- 商品属性存储更灵活，无需额外表结构变更
- 可暂时不引入Elasticsearch，降低系统复杂度
- 查询性能在大数据量下表现更好

### 负面影响
- 团队需要学习PostgreSQL特有语法
- 部分ORM工具支持不如MySQL完善
- 云数据库选型较少

### 风险
- 团队PostgreSQL运维经验不足，需培训

## 相关决策
- ADR-002: 使用Redis作为分布式缓存
```

### 示例2：微服务拆分决策
**输入**:
```
title: 订单服务与库存服务拆分决策
context: 原单体应用中订单和库存耦合严重，需要评估是否拆分为独立服务。
```

**输出**:
```markdown
# ADR-005: 订单服务与库存服务拆分决策

## 状态
Accepted

## 背景
当前订单和库存模块在同一个单体应用中：
1. 库存扣减逻辑在订单创建事务中
2. 库存回滚依赖订单事务回滚
3. 两个模块代码耦合度高
4. 库存查询性能成为瓶颈

业务发展需要：
- 库存服务需要独立部署支持多渠道
- 需要独立的库存预警和补货系统

## 决策
将订单服务和库存服务拆分为独立微服务。

拆分策略：
1. 库存服务独立部署
2. 使用最终一致性替代事务一致性
3. 通过消息队列实现库存异步扣减
4. 保留本地库存缓存应对高并发

## 备选方案
### 方案1: 保持单体架构
- 优点: 无分布式事务问题，开发简单
- 缺点: 无法满足多渠道需求，扩展性差
- 不选择原因: 业务发展需要独立库存服务

### 方案2: 服务拆分+强一致性
- 优点: 数据一致性强
- 缺点: 性能差，实现复杂
- 不选择原因: 性能不满足业务需求

## 影响
### 正面影响
- 库存服务可独立扩展
- 支持多渠道库存共享
- 降低单服务复杂度

### 负面影响
- 需要处理分布式事务问题
- 运维复杂度增加
- 开发联调成本上升

### 风险
- 数据一致性风险，需要完善的补偿机制

## 相关决策
- ADR-003: 使用RabbitMQ作为消息队列
- ADR-004: 最终一致性方案设计
```

## 注意事项
- ADR是记录而非讨论文档，应简洁明了
- 即使决策后来被证明是错误的，也应保留原始记录
- 每个决策只解决一个架构问题
- 决策状态变更时及时更新文档
- ADR编号按时间顺序递增，不可重复
